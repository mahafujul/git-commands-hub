üîµ Understanding Git & GitHub (Core Concepts)

What is a Remote?

Remote = A connection to a repository stored somewhere else (like GitHub or GitLab).

Think of it like a bookmark to another copy of your code that lives on the internet.

Common scenario:
  • Your code lives on your laptop (local)
  • A copy lives on GitHub (remote called "origin")
  • You push/pull to keep them in sync

Why Use Multiple Remotes?

In real-world projects, teams often connect to multiple remote repositories:

Example:
  • origin ‚Üí GitHub (primary collaboration)
  • gitlab ‚Üí GitLab (backup or internal access)
Benefits:
  • Code backup across platforms
  • Different access controls for different teams
  • Flexibility in collaboration workflows

Standard Remote Naming

Convention everyone follows:
  • origin ‚Üí Your primary remote (usually GitHub)
  • upstream ‚Üí Original project you forked from
  • gitlab ‚Üí Secondary remote (if using GitLab)
Following this convention makes commands predictable and easier to understand.

Working with Remotes - Basic Commands

git remote -vList all remotes connected to your repository
git remote add <name> <url>Connect to a new remote repository
git remote rename <old> <new>Rename an existing remote
git remote remove <name>Disconnect from a remote

Pushing to Different Remotes

git push origin mainPush to GitHub (origin)
git push gitlab mainPush to GitLab (gitlab)
git push --all originPush all branches to origin

üîê SSH Keys (Secure & Passwordless Access)

What is SSH and Why Should I Care?

SSH (Secure Shell) lets you connect to GitHub/GitLab without typing your password every time.

Without SSH:
  • Type username & password for every push
  • Less secure (passwords can be stolen)
  • Annoying and time-consuming
With SSH:
  • Push/pull automatically - no passwords!
  • More secure (uses cryptographic keys)
  • Industry standard for professional developers

How SSH Works (Simple Explanation)

SSH uses two keys (think of them like a lock and key):

  • Private key ‚Üí Stays on your computer (never share this!)
  • Public key ‚Üí You upload to GitHub/GitLab
When you push code, GitHub checks if your private key matches the public key you uploaded. If it matches, you're in!

One-Time Setup: Generate SSH Key

ssh-keygen -t ed25519 -C "your-email@example.com"Generate a new SSH key pair (modern & secure)
ls ~/.sshCheck that your keys were created successfully

Add SSH Key to SSH Agent

eval "$(ssh-agent -s)"Start the SSH agent
ssh-add ~/.ssh/id_ed25519Add your private key to the agent
The SSH agent manages your keys so you don't have to manually load them every time.

Think of it as a key manager that remembers your keys for you.

Upload Public Key to GitHub/GitLab

Follow these steps

# 1. Copy your public key to clipboard
cat ~/.ssh/id_ed25519.pub

# 2. Go to GitHub/GitLab Settings
#    GitHub: Settings ‚Üí SSH and GPG keys ‚Üí New SSH key
#    GitLab: Settings ‚Üí SSH Keys ‚Üí Add new key

# 3. Paste the key and give it a name (e.g., "My Laptop")

# 4. Save!

Test Your SSH Connection

ssh -T git@github.comTest connection to GitHub
ssh -T git@gitlab.comTest connection to GitLab
Success message: "Hi username! You've successfully authenticated..."
Error? Double-check you uploaded the correct public key.

Why Teams Use SSH

SSH is essential for:
  • Automated deployments (CI/CD pipelines)
  • Team collaboration without password sharing
  • Secure access to private repositories
  • Professional development workflows
Once set up, you'll never think about authentication again!

üéüÔ∏è Personal Access Tokens (PAT)

What is a PAT and Why Do I Need It?

Personal Access Token (PAT) is like a temporary password that gives specific permissions to apps and services.

Think of it as: A special key that unlocks only certain doors, not your entire account.

Why PATs exist:
  • GitHub disabled password authentication in 2021
  • More secure than passwords (you can limit what they can do)
  • Can be revoked without changing your password
  • Required for API access, automation, and Docker

PAT vs SSH: When to Use Which?

Use SSH for:
  • Git operations (push, pull, clone)
  • Daily development work
  • Personal repositories
Use PAT for:
  • GitHub API access
  • Docker image publishing (GHCR)
  • CI/CD pipelines (GitHub Actions)
  • Third-party apps and services
  • Automation scripts
Pro tip: You can use both! SSH for git, PAT for everything else.

Creating a Personal Access Token

Step-by-step guide

# 1. Go to GitHub Settings
#    Settings ‚Üí Developer settings ‚Üí Personal access tokens ‚Üí Tokens (classic)

# 2. Click "Generate new token (classic)"

# 3. Give it a descriptive name (e.g., "Docker GHCR Access")

# 4. Set expiration (recommended: 90 days for security)

# 5. Select scopes (permissions):
#    ‚úÖ repo (full control of private repos)
#    ‚úÖ write:packages (upload packages)
#    ‚úÖ read:packages (download packages)
#    ‚úÖ delete:packages (delete packages)

# 6. Click "Generate token"

# 7. COPY THE TOKEN IMMEDIATELY! You won't see it again!

Storing Your PAT Securely

export GITHUB_TOKEN="your_pat_here"Store PAT in environment variable (temporary)
echo "export GITHUB_TOKEN=your_pat_here" >> ~/.bashrcStore PAT permanently in bash profile
echo $GITHUB_TOKENVerify PAT is stored
‚ö†Ô∏è NEVER commit PATs to Git repositories!

Secure storage options:
  • Environment variables ‚Üí Store in .bashrc or .zshrc
  • Password manager ‚Üí 1Password, Bitwarden, LastPass
  • GitHub CLI ‚Üí Automatically stores credentials
  • Git Credential Manager ‚Üí Built-in credential storage

Using PAT for Git Operations (HTTPS)

git clone https://github.com/username/repo.gitClone with HTTPS (will prompt for PAT)
git remote set-url origin https://<PAT>@github.com/user/repo.gitAdd PAT directly to remote URL (not recommended)
git config --global credential.helper storeStore credentials permanently (use with caution)
When prompted:
  • Username: Your GitHub username
  • Password: Your PAT (not your actual password!)

Real-World Use: Docker Login with PAT

echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdinLogin to GitHub Container Registry using PAT
docker login ghcr.ioLogin interactively (will prompt for username and PAT)
GitHub Container Registry (GHCR) stores Docker images.
You need a PAT to authenticate and publish images.

Publishing Docker Images to GHCR

Complete workflow

# 1. Build your Docker image
docker build -t myapp:latest .

# 2. Tag image for GHCR (use lowercase username!)
docker tag myapp:latest ghcr.io/username/myapp:latest
docker tag myapp:latest ghcr.io/username/myapp:v1.0.0

# 3. Login to GHCR with PAT
echo $GITHUB_TOKEN | docker login ghcr.io -u username --password-stdin

# 4. Push image to GHCR
docker push ghcr.io/username/myapp:latest
docker push ghcr.io/username/myapp:v1.0.0

# 5. Make package public (if needed)
#    Go to: github.com/users/username/packages ‚Üí Package settings ‚Üí Change visibility

Docker Tagging Best Practices

docker tag myapp:latest ghcr.io/user/myapp:latestTag as latest version
docker tag myapp:latest ghcr.io/user/myapp:v1.2.3Tag with semantic version
docker tag myapp:latest ghcr.io/user/myapp:stableTag as stable release
Common tagging strategies:
  • latest ‚Üí Most recent version (auto-updated)
  • v1.0.0 ‚Üí Specific semantic version
  • stable ‚Üí Production-ready release
  • dev ‚Üí Development builds
  • sha-a1b2c3d ‚Üí Git commit SHA
Example: One image can have multiple tags

Pulling Images from GHCR

docker pull ghcr.io/username/myapp:latestPull latest version (public images)
echo $GITHUB_TOKEN | docker login ghcr.io -u username --password-stdinLogin first for private images
docker pull ghcr.io/username/myapp:v1.0.0Pull specific version

Managing Your PATs

gh auth statusCheck GitHub CLI authentication status
gh auth refreshRefresh GitHub CLI token
Security best practices:
  • Set expiration dates (30-90 days recommended)
  • Use minimum required permissions
  • Revoke tokens you're not using
  • Never share tokens or commit them to repos
  • Rotate tokens regularly
To revoke a PAT: GitHub Settings ‚Üí Developer settings ‚Üí Personal access tokens ‚Üí Delete

Troubleshooting PAT Issues

Common errors and solutions:
  • "403 Forbidden" ‚Üí Check token permissions (scopes)
  • "401 Unauthorized" ‚Üí Token expired or invalid
  • "denied: permission_denied" ‚Üí Need write:packages scope
  • "no basic auth credentials" ‚Üí Not logged in to Docker
